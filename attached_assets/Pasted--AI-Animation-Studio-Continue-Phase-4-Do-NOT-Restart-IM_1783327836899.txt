## Architecture

The Knowledge module uses the prefix:

kn_

Models are intentionally consolidated instead of creating unnecessary tables.

Current planned models:

* KnowledgeCollection
* KnowledgeDocument
* KnowledgeChunk
* EmbeddingJob
* KnowledgeMemory
* RetrievalHistory
* KnowledgeVersion

Instead of separate tables:

* KnowledgeSource
* KnowledgeMetadata
* ContextSnapshot
* VectorIndex

their information should be stored inside JSON metadata fields where appropriate.

This is an intentional production design decision.

---

## Embeddings

Create an EmbeddingProvider abstraction.

Support:

* MockEmbeddingProvider
* OllamaEmbeddingProvider

Configuration:

EMBEDDING_PROVIDER

Values:

* mock
* ollama

Unknown providers must safely fall back to mock.

Follow exactly the same provider architecture used for Story Intelligence.

---

## Vector Store

Create a VectorStoreProvider abstraction.

Support:

Default:

* In-memory deterministic vector search

Optional:

* ChromaDB

Future:

* Qdrant

Configuration:

VECTOR_DB_PROVIDER

Values:

* memory
* chromadb
* qdrant

Do not hardcode providers.

---

## Embeddings

KnowledgeChunk stores embedding vectors directly.

Use JSON float arrays.

Do not depend on pgvector.

Cosine similarity should be implemented in pure Python.

---
