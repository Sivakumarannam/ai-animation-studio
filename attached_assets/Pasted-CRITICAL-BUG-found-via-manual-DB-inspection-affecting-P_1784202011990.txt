CRITICAL BUG — found via manual DB inspection, affecting Phases 3, 4, and 5.

Root cause: backend/database/connection.py's get_session() is an async
generator with commit-after-yield semantics:

  async def get_session():
      async with _session_factory() as session:
          try:
              yield session
              await session.commit()
          except Exception:
              await session.rollback()
              raise

This is the correct pattern for FastAPI's Depends() injection, where the
framework fully drains the generator on request completion. But it is being
used incorrectly in Celery task "core" functions with this shape:

  async for session in get_session():
      ...
      return result   # <- return INSIDE the loop body

Returning from inside a loop over an async generator forces Python to call
.aclose() on it, which throws GeneratorExit AT the yield line — meaning
`await session.commit()` (the line after yield) NEVER executes. The
function returns a fully-formed, convincing result, all service calls run
normally, all logging fires — but the transaction is silently discarded
every time.

CONFIRMED VIA DIRECT DATABASE INSPECTION: triggered
POST /api/v1/si/projects/{id}/generate twice today. Celery logged full
pipeline_complete success both times with real-looking world_id, season_id,
episode_id, scene_count=5. Direct psql query on cartoon_studio confirmed
ZERO rows exist for either episode_id in si_episodes, si_worlds, si_seasons,
or si_story_scenes. Nothing was ever persisted, despite 100% "successful"
task_success logs.

AFFECTED FILES — confirmed via grep, same broken pattern in every core
function of all three files:
- backend/apps/worker/tasks/intelligence_tasks.py
  (_run_full_pipeline_core, _generate_episode_core)

  (6 core functions, lines ~56, ~94, ~140, ~174, ~211, ~247)

FIX — for every one of these 10 functions, add an explicit commit
immediately before the early return, inside the loop, so correctness does
not depend on the generator's post-yield code ever executing:

  async for session in get_session():
      ...
      result = await orchestrator.some_method(...)
      await session.commit()
      return result
  return {}

Do this for all 10 functions across all 3 files. Do NOT change
get_session() itself — it is correct for its intended FastAPI Depends()
use case; the bug is entirely in how these Celery tasks consume it.
