CRITICAL LESSON FROM TODAY — READ FIRST:
A serious bug was just found and fixed across intelligence_tasks.py,
knowledge_tasks.py, and research_tasks.py: 10 Celery core functions used
the pattern `async for session in get_session(): ... return result` — an
early return inside the loop forces the async generator to close via
GeneratorExit AT the yield line, meaning the commit-after-yield code in
get_session() (backend/database/connection.py) never runs. Every one of
these functions logged full "success" while silently never persisting
anything — confirmed via direct psql inspection showing zero rows for
episodes that Celery had logged as fully created.

The fix (already applied elsewhere): call `await session.commit()`
explicitly INSIDE the loop, immediately before the return, e.g.:
  async for session in get_session():
      ...
      result = await orchestrator.some_method(...)
      await session.commit()
      return result
  return {}

asset_tasks.py, animation_tasks.py, and voice_tasks.py (Phases 6-8) already
use the CORRECT pattern via session_scope() — use THAT pattern as your
template for Phase 9's Celery tasks, not the broken get_session() loop
pattern. Before considering Phase 9 complete, explicitly grep your own new
task file for this anti-pattern:

This should return NOTHING. If it does, you've reintroduced the bug — fix
it before reporting complete.
