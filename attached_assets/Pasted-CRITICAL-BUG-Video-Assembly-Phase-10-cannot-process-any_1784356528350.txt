mock outputs due to a storage_key convention mismatch, discovered via

prior bugs — task_kwargs shape and _noop() sync fallback — are fixed and
confirmed working). But the task fails with an FFmpeg error trying to open:
  animations/mock/{project_id}/scene_905632ec-...-0b4260be.mp4
as if it were a literal local filesystem path.

ROOT CAUSE: video_assembly_service.py assumes mock outputs are identifiable
by a "mock://" URI prefix:
  if o.storage_key and not o.storage_key.startswith("mock://"):
But Phase 7's animation mock provider (and likely Phase 8/9's) actually
generates keys like "animations/mock/{project}/scene_{id}_{hash}.mp4" —
using "mock" as a PATH SEGMENT, not a URI scheme. Since this doesn't start
with literal "mock://", the assembly service incorrectly treats it as a
real downloadable/local file.
